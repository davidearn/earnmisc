# AGENTS.md for earnmisc

This file applies to the `earnmisc` package repository.

It is package-specific. In the combined `pkg-devel` project, also follow the root-level `pkg-devel/AGENTS.md` for shared workflow, logging, Git, verification, and general R-package conventions.

Do not edit this file unless the user explicitly asks you to edit it.

## Package purpose

`earnmisc` is a small personal R utility package for general-purpose helper functions used across other packages, including `gaemr`, `agemortr`, `mcmarksr2`, and likely additional downstream packages in the future.

The package should remain small, clean, well-documented, minimally dependent, and focused on reusable utilities rather than project-specific scientific code.

`earnmisc` is the natural home for generic helpers that multiple packages can use. Downstream packages should call `earnmisc` utilities rather than carrying local copies when the helper is genuinely general.

## Package scope

Current intended utility families include:

- Okabe--Ito colour palette helpers and exported colour constants.
- Base-graphics helpers, including `xys_line()` and the `mts_*` plotting helpers.
- TeX-aware plot-label helpers via `nice_text()`.
- Tikz-device workflow helpers, including `tikz_open()`, `tikz_info()`, and `tikz_compile()`.
- List/object utilities such as `update_list()` and `input_form()`.
- Device-detection helpers such as `dev_is_tikz()` and `dev_is_pdf()`.

Do not add project-specific epidemic-model or manuscript-specific scientific functions to `earnmisc`.

Before exporting an additional helper, decide whether it is genuinely general-purpose enough to belong here.

## Okabe--Ito colour helpers

Okabe--Ito helpers should provide the standard colourblind-friendly palette with clear names such as:

```r
black
orange
sky_blue
bluish_green
yellow
blue
vermillion
reddish_purple
```

The extended palette is the default, but documentation should distinguish the original Okabe--Ito palette from extended convenience colours.

The intended plotting idiom for exported colour constants is:

```r
col = oi.orange
```

or, with alpha adjustment:

```r
col = oi_alpha(oi.orange, 0.3)
col = oi_colour("orange", alpha = 0.3)
```

Do not support quoted pseudo-colour names such as:

```r
col = "oi.orange"
```

## `xys_line()` design

`xys_line()` is an S3 generic with dispatch formal `object`:

```r
xys_line <- function(object, ...) {
  UseMethod("xys_line")
}
```

The dispatch formal is `object` so that the ordinary point/slope API can still use `x` as the x-coordinate.

The default numeric method should support the user-facing API:

```r
xys_line(x, y, slope, ...)
xys_line(x = 0, y = 0, slope = 1)
```

For finite slopes, it should draw the line equivalent to:

```r
graphics::abline(a = y - slope * x, b = slope, ...)
```

For infinite slopes, it should draw a vertical line at `x`, equivalent to:

```r
graphics::abline(v = x, ...)
```

It should support:

- scalar and vectorised `x`, `y`, and `slope`;
- all-combinations expansion when more than one coordinate input is vectorised;
- vectorised graphical parameters such as `col`, `lty`, and `lwd`;
- scalar invisible return as `c(intercept = ..., slope = ...)` for one finite line;
- data-frame invisible return for multiple lines.

Downstream S3 methods should use `object` as the first formal. Coordinate arguments should follow as `x`, `y`, and `slope`.

## `mts_*` plotting helpers

Use the `mts_*` names only:

```r
mts_plot()
mts_lines()
mts_abline()
mts_legend()
mts_set_panel()
mts_plot_overlay()
mts_xys_line()
```

Do not use or reintroduce these old names:

```r
plot_mts()
lines_mts()
abline_mts()
legend_mts()
set_mts_panel()
plot_mts_overlay()
add_to_mts_panel()
with_mts_panel()
```

If a future helper is added for adding annotations to an existing panel without clearing it, call it:

```r
mts_add_to_panel()
```

`mts_plot()` creates the controlled multi-panel layout and stores an `earnmisc_mts_plot_info` object. Important metadata includes:

```r
blank.panels
data.panels
panel.roles
panels
curves
```

`mts_lines()`, `mts_abline()`, `mts_xys_line()`, `mts_set_panel()`, and `mts_legend()` are designed for layouts created by `mts_plot()`. Arbitrary existing `plot.mts()` output is intentionally unsupported.

`mts_set_panel()` clears and reinitialises a selected panel. It is intended mainly for reserved blank panels, legends, notes, and custom panel content.

`mts_plot_overlay()` is a convenience wrapper around `mts_plot()` plus repeated `mts_lines()` calls. It does not draw legends automatically.

`mts_legend()` uses the `plot.info$curves` registry. Placement policy:

- if `panel` is supplied, draw in that full layout panel;
- if `panel = NULL`, use the first reserved blank panel;
- if no panel is supplied and no blank panel exists, error clearly.

Legend grouping should support source, column, and curve groupings. Preserve expression-like source labels and `nice_text()` output through the `source.label` list-column.

## `nice_text()` design

`nice_text()` is a lightweight plot-label helper, not a full TeX parser.

It should support tikz and non-tikz workflows:

- `use.tikz = TRUE`: expand package/user macros and return character LaTeX strings; do not call `latex2exp::TeX()`.
- `use.tikz = FALSE`: expand macros, clean ignored TeX commands, and call `latex2exp::TeX()` when available.
- `use.tikz = NULL`: look for a scalar logical `use.tikz` in the calling environment; otherwise default to `FALSE`.

Macro expansion applies in both tikz and non-tikz modes.

If `latex2exp` is unavailable in non-tikz mode, return the preprocessed character vector rather than failing.

User files may append to or replace package defaults through function arguments and package options. The parser is intentionally limited to simple no-argument macros.

## Tikz helpers

`tikz_open()` wraps `tikzDevice::tikz()` conditionally. Keep `tikzDevice` in `Suggests`, not `Imports`, unless there is a deliberate package design change.

`tikz_open()` should:

- default to `standAlone = TRUE`;
- expose important `tikzDevice::tikz()` arguments explicitly;
- store useful metadata retrievable by `tikz_info()`;
- return that metadata invisibly;
- print a helpful opening message by default;
- not print `done`, because the device is not closed by `tikz_open()`.

`tikz_compile()` should accept either a `.tex` filename or a `tikz_open()` info-like object, compile in the `.tex` file directory, default to `lualatex` batch mode, return the produced PDF filename, and error clearly if compilation fails.

Do not require a working LaTeX installation, `tikzDevice`, or `latex2exp` for ordinary checks. Tests requiring optional tools must skip cleanly.

## List and input-form utilities

`update_list()` supports top-level updates and simple `$`-separated nested paths, for example:

```r
update_list(x, type = "new")
update_list(x, "parms$graphics$lwd" = 3)
```

`input_form()` produces pasteable R input using `deparse()` and supports formatting/file-output controls including `prefix`, `suffix`, `final.newline`, `overwrite`, and `align`.

Important `input_form()` behaviours:

- `align = NULL`: ordinary deparse output;
- `align = ","`: leading comma style for suitable named lists;
- `align = c(",", "=")`: leading commas and aligned equals signs;
- `overwrite = TRUE`, `"warn"`, `"recover"`, `"error"`;
- `FALSE` is a synonym for `"error"`;
- `overwrite = "recover"` creates `.bak`, `.bak1`, etc. before overwriting.

## Device helpers

`dev_is_tikz()` and `dev_is_pdf()` should return scalar logical values based on the current graphics device.

`dev_is_tikz()` may use the device name and/or `tikz_open()` metadata. `dev_is_pdf()` checks for the PDF device name.

## Package-specific caution notes

Keep `earnmisc` generic and dependency-light.

Do not make `earnmisc` depend on downstream packages such as `gaemr`.

If a downstream package needs behaviour that is only meaningful for that downstream package's objects, implement the S3 method or adapter downstream, not in `earnmisc`.

If a generic helper is useful across packages, implement the general reusable part here first and let downstream packages call it.
